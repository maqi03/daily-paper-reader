## 摘要
使用大语言模型（LLMs）自动化优化建模是实现可扩展决策智能的一条充满前景的路径，但现有方法要么依赖于基于高推理延迟的闭源 LLM 构建的智能体流水线，要么使用昂贵的过程监督（process supervision）微调较小的 LLM，而这往往会过拟合到单一的求解器 API。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种执行验证学习框架，将数学规划求解器视为确定性的交互式验证器。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的表述消除了对过程级监督的需求，并通过切换验证环境（而非重建特定于求解器的数据集）实现了跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 数据集上，针对 Gurobi、OR-Tools 和 COPT 的实验表明，EVOM 的表现达到或超过了经过过程监督微调（SFT）的模型，支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效且低成本的求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

    1.  **成本高昂**：现有的微调方法（SFT）高度依赖“过程监督”，需要昂贵的中间推理步骤和参考代码标注。

*   **核心目标**：利用数学求解器作为“天然验证器”的特性，通过强化学习（RL）实现仅需结果监督（Outcome-only）的自动化建模。

### 2. 方法论：EVOM 框架
论文提出了 **EVOM（Execution-Verified Optimization Modeling）**，其核心思想是**将求解器视为确定性的交互式验证器**。
*   **闭环流程**：采用“生成-执行-反馈-更新”的循环。
    1.  **生成**：模型根据问题和目标求解器标识，生成包含思考过程（`<think>`）和代码（`<code>`）的响应。
    2.  **执行**：在沙箱环境中调用目标求解器执行代码，获取执行状态（是否报错）、求解状态（是否最优）和目标函数值。
    3.  **反馈（奖励设计）**：
        *   **格式奖励**：检查输出是否符合特定的 XML 标签格式。
        *   **结果正确性奖励**：对比求解器输出的目标值与真实答案。若数值接近或状态一致，则给予正奖励。
    4.  **更新**：使用 **GRPO**（群体相对策略优化）或 **DAPO** 算法，通过群体内的相对优势来更新策略，无需额外的 Critic 模型。
*   **求解器条件化**：在 Prompt 中加入求解器名称，使模型能够根据不同环境生成对应的 API 代码。

### 6. 主要结论与发现
*   **结果监督胜过过程监督**：仅靠求解器反馈的 RL 训练，效果即可达到或超过依赖昂贵人工标注的 SFT 方法。

### 7. 优点
*   **低成本可扩展**：摆脱了对中间推理步骤标注的依赖，只需“问题-答案”对即可训练。
*   **环境自适应**：通过“环境切换”而非“数据重构”来适配新求解器，极大地降低了工业迁移成本。
*   **鲁棒性强**：实验证明该方法对优化算法（GRPO vs DAPO）不敏感，核心增益来自于求解器的确定性反馈。

### 8. 不足与局限
*   **冷启动挑战**：对于模型完全未见过的求解器 API，如果初始成功率为 0，RL 无法起步（论文通过少量 SFT 缓解了此问题）。
*   **逻辑深度限制**：错误分析显示，虽然 RL 显著减少了语法和变量类型错误，但在处理极其复杂的逻辑约束（Constraint Error）方面，提升幅度相对较小。
*   **反馈稀疏性**：对于非常复杂的问题，随机采样可能很难获得正向奖励，可能需要更精细的课程学习或分步奖励设计。
