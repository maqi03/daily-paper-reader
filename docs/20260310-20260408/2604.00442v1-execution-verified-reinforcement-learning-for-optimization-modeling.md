## 摘要
利用大语言模型（LLMs）实现优化建模自动化是通往可扩展决策智能的一条极具前景的路径，但现有方法要么依赖于基于闭源 LLM 构建的、具有高推理延迟的智能体流水线，要么使用昂贵的、且往往会过拟合于单一求解器 API 的过程监督来微调较小的 LLM。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种将数学规划求解器视为确定性交互式验证器的执行验证学习框架。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的公式化方法消除了对过程级监督的需求，并通过切换验证环境而非重建特定于求解器的数据集，实现了跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 数据集上，针对 Gurobi、OR-Tools 和 COPT 求解器的实验表明，EVOM 的表现达到或超过了经过过程监督微调（SFT）的模型，支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效的低成本求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

### 2. 方法论：核心思想与技术细节
EVOM 采用“生成-执行-反馈-更新”的闭环学习流程：
*   **核心思想**：将求解器视为环境，将执行结果（编译是否通过、解是否可行、目标值是否匹配）转化为奖励信号，无需人工标注中间推理过程。
*   **关键技术细节**：
    *   **求解器条件化（Solver-Conditioning）**：在 Prompt 中指定目标求解器（如 Gurobi, OR-Tools），要求模型生成特定 API 的代码。
    *   **沙箱执行系统**：在受限环境下运行生成的代码，捕获执行状态（成功/失败）、求解状态（最优/不可行/无界）和目标函数值。
    *   **奖励函数设计**：
        *   **格式奖励（Format Reward）**：通过正则匹配确保模型遵循 `<think>` 和 `<code>` 的结构。
        *   **结果奖励（Outcome Reward）**：如果执行成功且得到的目标值与真实答案（Ground Truth）在误差范围内一致，则给予高奖励。
    *   **策略优化算法**：使用 **GRPO**（组相对策略优化）和 **DAPO**（解耦剪切与动态采样）进行训练。GRPO 通过组内平均值计算优势函数，无需额外的 Critic 模型，适合稀疏的可验证奖励场景。

### 6. 主要结论与发现
*   **性能卓越**：仅使用结果监督（Outcome-only）的 EVOM 在多个指标上达到或超过了使用昂贵过程监督的 SFT 模型。

*   **推理能力的涌现**：强化学习诱导模型产生了更长的思考过程（Thinking length 增加），且代码更加简洁。
*   **低成本适配**：对于新求解器，只需切换执行环境并进行少量 RL 训练即可快速适配，无需重新构建大规模标注数据集。

### 7. 优点（亮点）
*   **摆脱数据依赖**：解决了优化建模中“高质量中间过程标注难获得”的痛点。
*   **解耦逻辑与语法**：通过 RL 学习到了通用的数学建模逻辑，而非死记硬背特定 API 的调用方式。
*   **闭环验证**：利用求解器的确定性反馈，确保了生成代码的可执行性和结果的准确性。

### 8. 不足与局限
*   **复杂逻辑瓶颈**：错误分析显示，虽然 RL 显著减少了语法和变量类型错误，但在处理极其复杂的数学约束逻辑（Deep Semantic Errors）时，提升幅度相对较小。
*   **冷启动依赖**：对于模型完全未见过的求解器（如 COPT），仍需极少量（约 100 条）的 SFT 数据进行初步引导，无法完全从零开始。
*   **计算开销**：RL 训练过程中的采样和多次求解器调用比传统的 SFT 更耗费计算资源。
